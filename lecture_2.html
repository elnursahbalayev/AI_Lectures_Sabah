1. From Optimization to Adversarial Thinking

Last week, we minimized \(f(x) = x^2 - 4x + 6\). The function was static — it didn't change its shape based on our moves. This is the world of single-agent optimization.

But the real world is full of adversarial environments: Chess opponents block your strategy. Cyber attackers adapt to your defenses. In GANs (Week 13), one neural network actively tries to fool another.

Week 1 — Single-Agent
\(\arg\min_\theta \mathcal{L}(\theta)\)
↓
Minimize loss, alone.

The landscape is fixed. We search for its lowest point.

Week 2 — Adversarial
\(\arg\max_{me} \min_{opponent}\)
↓
Maximize my outcome,
assuming opponent minimizes it.

The landscape changes with every opponent move.

"Intelligence isn't just about optimizing alone — it's about optimizing when someone is actively trying to beat you."

2. Formalizing Games

A formal game (in the AI sense) is defined by:

  • Players: Typically two — MAX (us) and MIN (opponent).
  • States: All possible configurations (e.g., all board positions).
  • Actions: Legal moves from any state.
  • Transition Function: \(S' = \text{Result}(S, a)\) — the state after action \(a\).
  • Terminal Test: \(\text{Terminal}(S) \in \{\text{True}, \text{False}\}\)
  • Utility Function: \(U(S)\) — numeric score for terminal states. Win → \(+1\), Loss → \(-1\), Draw → \(0\).

The Game Tree

A Game Tree is a tree where the root is the current state, each edge is a legal move, levels alternate between MAX and MIN, and leaves are terminal states with known utility values.

Tic-Tac-Toe

~255,168 possible games. Fully solvable — we can search the entire tree.

Chess

\(\approx 10^{120}\) possible games. More than atoms in the observable universe. Not solvable by brute force.

Go

\(\approx 10^{360}\) possible games. Required Monte Carlo Tree Search + Deep RL (AlphaGo, 2016).

Dota 2

Continuous actions, partial info, 5v5 teamwork. OpenAI Five (2019) used pure Reinforcement Learning.

3. The Minimax Algorithm

Minimax is the foundational algorithm for adversarial search. The logic is beautifully simple:

"MAX assumes MIN will always play optimally (worst case for MAX). MAX then chooses the move that maximizes the outcome under this worst-case assumption."

The Recursive Definition

\[\text{minimax}(S) = \begin{cases} U(S) & \text{if } S \text{ is terminal} \\ \max_{a} \text{minimax}(\text{Result}(S, a)) & \text{if MAX's turn} \\ \min_{a} \text{minimax}(\text{Result}(S, a)) & \text{if MIN's turn} \end{cases}\]

The Algorithm in Code

# ============================================================
# MINIMAX ALGORITHM
# This function recursively evaluates every possible future
# game state and returns the best achievable score.
#
# Parameters:
#   state          - the current game state (e.g., board config)
#   is_maximizing  - True if it's MAX's turn (us),
#                    False if it's MIN's turn (opponent)
#
# Returns:
#   The optimal utility value (integer score) for this state.
# ============================================================
def minimax(state, is_maximizing):

    # ------- BASE CASE -------
    # If the game is over (win / loss / draw), return the
    # actual score.  No more moves to explore.
    if is_terminal(state):
        return utility(state)   # e.g. +1 (win), -1 (loss), 0 (draw)

    # ------- MAX's TURN (our turn) -------
    # We want to MAXIMIZE the score.
    # Start with the worst possible value (-infinity) and
    # try every legal move looking for something better.
    if is_maximizing:
        best = -infinity                    # worst case for MAX
        for action in get_actions(state):   # loop over every legal move
            child = result(state, action)   # apply the move → new state
            value = minimax(child, False)   # recurse: now it's MIN's turn
            best = max(best, value)         # keep the highest score found
        return best                         # return the best score MAX can get

    # ------- MIN's TURN (opponent's turn) -------
    # The opponent wants to MINIMIZE our score.
    # Start with the best possible value (+infinity) and
    # try every legal move looking for something lower.
    else:
        best = +infinity                    # worst case for MIN
        for action in get_actions(state):   # loop over every legal move
            child = result(state, action)   # apply the move → new state
            value = minimax(child, True)    # recurse: now it's MAX's turn
            best = min(best, value)         # keep the lowest score found
        return best                         # return the best score MIN can get

Complexity Analysis

Time Complexity

\(O(b^m)\) where \(b\) = branching factor, \(m\) = maximum depth. Exponential — scales terribly.

Space Complexity

\(O(b \cdot m)\) — only the current path needs to be in memory (depth-first search).

The Exponential Wall:
  • Tic-Tac-Toe: \(b \approx 5\), \(m = 9\) → ~2 million nodes ✓
  • Chess: \(b \approx 35\), \(m \approx 80\) → \(10^{120}\) nodes ✗
  • Go: \(b \approx 250\), \(m \approx 150\) → \(10^{360}\) nodes ✗

4. Alpha-Beta Pruning

Alpha-Beta Pruning is one of the most elegant ideas in computer science. It produces the exact same result as Minimax but skips evaluating branches that cannot possibly influence the final decision.

The Key Insight

During search, we maintain two bounds:

  • \(\alpha\) = the best (highest) value MAX can guarantee so far.
  • \(\beta\) = the best (lowest) value MIN can guarantee so far.
Pruning Rule: If at any point \(\alpha \geq \beta\), stop evaluating the current subtree. The remaining branches are irrelevant — one player already has a better option elsewhere.

The Pruned Algorithm

# ============================================================
# ALPHA-BETA PRUNING
# An optimized version of Minimax that skips ("prunes")
# branches that CANNOT affect the final decision.
# It produces the EXACT SAME result as plain Minimax but
# is much faster because it evaluates fewer nodes.
#
# Parameters:
#   state          - current game state
#   alpha (α)      - best score MAX can guarantee so far
#                    (starts at -infinity)
#   beta  (β)      - best score MIN can guarantee so far
#                    (starts at +infinity)
#   is_maximizing  - True if MAX's turn, False if MIN's turn
#
# KEY RULE: If α >= β, we PRUNE (stop exploring) the
# current branch because one player already has a
# better option elsewhere in the tree.
# ============================================================
def alpha_beta(state, alpha, beta, is_maximizing):

    # Base case: game is over → return the actual score
    if is_terminal(state):
        return utility(state)

    # ------- MAX's TURN -------
    if is_maximizing:
        value = -infinity                   # start with the worst for MAX
        for action in get_actions(state):   # try each legal move
            child = result(state, action)   # get the resulting state

            # Recurse into opponent's turn, passing α and β down
            value = max(value, alpha_beta(child, alpha, beta, False))

            # Update α: MAX now knows it can guarantee at least 'value'
            alpha = max(alpha, value)

            # ✂ PRUNE CHECK (β cutoff):
            # If α >= β, MIN already has a better path elsewhere,
            # so MIN would NEVER let us reach this branch.
            # No point exploring remaining children → break out.
            if alpha >= beta:
                break  # β cutoff — prune remaining siblings
        return value

    # ------- MIN's TURN -------
    else:
        value = +infinity                   # start with the worst for MIN
        for action in get_actions(state):   # try each legal move
            child = result(state, action)   # get the resulting state

            # Recurse into MAX's turn, passing α and β down
            value = min(value, alpha_beta(child, alpha, beta, True))

            # Update β: MIN now knows it can guarantee at most 'value'
            beta = min(beta, value)

            # ✂ PRUNE CHECK (α cutoff):
            # If α >= β, MAX already has a better path elsewhere,
            # so MAX would NEVER choose this branch.
            # No point exploring remaining children → break out.
            if alpha >= beta:
                break  # α cutoff — prune remaining siblings
        return value

How Much Does It Save?

With perfect move ordering (best moves first):

\[\text{Minimax: } O(b^m) \qquad \longrightarrow \qquad \text{Alpha-Beta: } O(b^{m/2})\]

This effectively doubles the searchable depth for the same computation budget. It's the difference between looking 6 moves ahead and looking 12 moves ahead in Chess.

5. Evaluation Functions

For complex games like Chess, even Alpha-Beta can't reach terminal states. The solution: cut off search at a fixed depth and estimate the value of non-terminal states with an Evaluation Function \(\text{Eval}(S)\).

What Makes a Good Evaluation Function?

  1. Fast to compute — it's called millions of times.
  2. Correlated with true utility — high eval should mean a likely win.
  3. Orders states correctly — \(\text{Eval}(S_1) > \text{Eval}(S_2)\) should mean \(S_1\) is truly better.

Example: Chess Evaluation

\[\text{Eval}(S) = w_1 \cdot \text{Material} + w_2 \cdot \text{Mobility} + w_3 \cdot \text{King Safety} + w_4 \cdot \text{Center Control}\]

Material

Piece values: Queen=9, Rook=5, Bishop/Knight=3, Pawn=1. Sum yours minus opponent's.

Mobility

Count of legal moves available. More options = stronger position.

King Safety

Is the king exposed? Castled? Pawn shield intact?

Center Control

How many central squares (d4, d5, e4, e5) are controlled?

The ML Connection: This evaluation function is a weighted sum — the exact structure as a linear model! Modern chess engines (Stockfish NNUE, AlphaZero) replaced hand-crafted eval with neural networks trained on millions of games. Game AI and Machine Learning converge.

Depth-Limited Minimax

# ============================================================
# DEPTH-LIMITED MINIMAX
# In complex games (like Chess), the game tree is far too
# large to search all the way to terminal states.
# Solution: search only up to a fixed DEPTH, then use a
# heuristic evaluation function to ESTIMATE the value of
# the state we stopped at.
#
# Parameters:
#   state          - current game state
#   depth          - how many more levels we are allowed to
#                    search (decreases by 1 each level)
#   is_maximizing  - True if MAX's turn, False if MIN's turn
#
# Returns:
#   The estimated utility value for this state.
# ============================================================
def minimax_depth_limited(state, depth, is_maximizing):

    # Base case 1: game is actually over → use real utility
    if is_terminal(state):
        return utility(state)

    # Base case 2: we've reached our search depth limit
    # Instead of exploring further, use a HEURISTIC function
    # to estimate how good this state is (e.g., piece count
    # in chess, mobility score, etc.)
    if depth == 0:
        return evaluate(state)  # heuristic estimate, NOT exact

    # ------- MAX's TURN -------
    if is_maximizing:
        best = -infinity                    # worst case for MAX
        for action in get_actions(state):   # try every legal move
            child = result(state, action)   # apply the move
            # Recurse with depth - 1 (one level closer to limit)
            value = minimax_depth_limited(child, depth - 1, False)
            best = max(best, value)         # keep the highest score
        return best

    # ------- MIN's TURN -------
    else:
        best = +infinity                    # worst case for MIN
        for action in get_actions(state):   # try every legal move
            child = result(state, action)   # apply the move
            # Recurse with depth - 1 (one level closer to limit)
            value = minimax_depth_limited(child, depth - 1, True)
            best = min(best, value)         # keep the lowest score
        return best

6. Introduction to Agents

An Agent is an entity that perceives its environment, decides on an action, and acts on the environment.

Environment → [Sensors] → Agent Logic → [Actuators] → Environment
(Minimax, ML model, Rule-based, etc.)

Types of Agents

Agent TypeHow It DecidesExample
Reflexif condition → actionThermostat
Model-BasedMaintains internal stateSelf-driving car
Goal-BasedPlans toward a goalMinimax game player
Utility-BasedMaximizes a utility functionRecommendation system
LearningImproves from experienceAlphaGo, ChatGPT

Our Tic-Tac-Toe bot in the tutorial is a Goal-Based Agent: its goal is to win (or not lose), and it uses Minimax to plan actions toward that goal.

From Games to the Real World

Preview: Each relaxation requires more sophisticated algorithms — Expectiminimax for randomness, POMDP for partial observability, MCTS for huge spaces. We revisit these in Week 14: Reinforcement Learning.

7. Historical Context: Games as AI's Testing Ground

Games have served as the benchmark for AI progress since the very beginning of the field.

YearMilestoneAlgorithm
